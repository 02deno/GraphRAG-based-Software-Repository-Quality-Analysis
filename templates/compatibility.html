    What will the knowledge graph contain? (node & edge types)

    The pipeline builds a directed graph: every edge goes from a source node id to a target node id. Below is what each implemented type means in this project (your results page will list which types actually appeared in your run).

    Node types (implemented)

    • File — one node per Python file under the repo (id is the repo-relative path).
    • Function — a function or method; linked to its file via IN_FILE.
    • Class — a class definition; linked to its file via IN_FILE.
    • Test — a test case/fixture extracted from test files; TESTS edges point to what it exercises.

    Edge types (implemented)

    IMPORTS — File → File. The source file’s import statements resolve to another module file in this repo.

    IN_FILE — Function or Class → File. The symbol lives inside that file (nests code structure under files).

    CALLS — Function → Function or Class. Static call-sites inside function bodies are linked to known symbols by name.

    TESTS — Test → Function or Class. The test is inferred to cover or reference that symbol (static link, not runtime coverage).

    The JSON schema also reserves other kinds (e.g. Commit) for future work; they are not emitted by the current Python extractor.
